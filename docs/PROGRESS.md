**Key architectural decisions (current — supersede earlier drafts):**
- Booking flow: single admin-panel-first flow (the earlier "Path B" GCal-first flow was removed) — PostgreSQL is the single source of truth, Google Calendar is a write-only mirror
- Session/package tracking: `client_packages` table with computed `sessions_remaining`, FIFO selection by exact `package_id`
- No-show handling: `no_show` booking status (counts as consumed), reinstated by editing the same booking record
- Security: **no field-level encryption** (an earlier draft mentioned AES-256 via prisma-field-encryption — that decision was reversed). Relies on bcrypt passwords + Neon at-rest encryption + access control
- Environments: dev/uat/main — 3 Neon DB branches + matching git branches; `uat` is periodically reset from `main` (Neon "reset from parent") then anonymized via a script
